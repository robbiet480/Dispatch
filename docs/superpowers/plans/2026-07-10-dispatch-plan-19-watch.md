# Dispatch Plan 19: Apple Watch app

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** an independent (companion-style) watchOS app — quick Yes/No answering and report filing from the wrist WITH watch-side sensor capture and device provenance, complications via WidgetKit accessory widgets, prompt notifications on the watch via automatic forwarding — synced through the existing private CloudKit database.

**Design doc (read first, it carries the citations and the tradeoff record):** `docs/superpowers/specs/2026-07-10-watch-app-design.md`. Tracks GitHub issue #7 — reference it in commits/PRs; do NOT close it from this plan (it covers the whole watch effort).

## Design decisions (decide + log)

- **Independent watchOS app + CloudKit, NOT WatchConnectivity.** The watch is a third device in the already-hardened sync mesh: own local SwiftData store in the watch-side app-group container, `cloudKitDatabase: .private(SyncPolicy.containerIdentifier)`. WatchConnectivity is not used anywhere in v1 (Apple: independent apps "can't rely on the Watch Connectivity framework"; CloudKit is the named sync path — see design doc citations).
- **App Groups are per-device.** The watch app + watch widget extension share the *watch's* `group.io.robbie.Dispatch` container; nothing is shared with the phone through it. No code or comment may imply cross-device container sharing. The watch runs neither `RemoteChangeObserver` nor SyncDedupe — the phone stays pipeline authority.
- **Device provenance on every report (user request, 2026-07-10).** `Report` gains OPTIONAL `sourceDeviceModel: String?` (raw hardware identifier via `utsname.machine`, e.g. `"iPhone17,1"`/`"Watch7,4"` — unentitled on both platforms) and `sourceDeviceName: String?` (`UIDevice.current.name` / `WKInterfaceDevice.current().name`). Additive, nil on all pre-existing reports, no migration. Stamped at creation time by the shared filing path on BOTH platforms — every trigger gets provenance. **Doc-cited reality:** since iOS 16 / watchOS 9 the name APIs return the GENERIC device name without `com.apple.developer.device-information.user-assigned-device-name`; **we are NOT requesting that entitlement now** (Apple approval + pinned-profile regen) — the field exists so real names can arrive later without a format change (citations in design §v1-scope-2). v2 export: both fields additive-optional, encode-when-present/decode-leniently, round-trip AND absence-tolerance tests.
- **Watch-side sensor capture (user amendment, 2026-07-10).** Watch filings run the SAME `CaptureCoordinator`/`ReportBuilder` flow as the phone — same 10s timeout, same partial-result semantics, capture never blocks filing. Watch provider set (availability doc-cited in design §v1-scope-3): location(+altitude) via CoreLocation, weather via WeatherKit (watchOS 9+), battery via `WKInterfaceDevice` + `isBatteryMonitoringEnabled` (watchOS 4+), all health kinds via HealthKit (watchOS 2+, native). **Phone-only, explicitly nil on watch reports:** photos (framework absent on watchOS), audio, connection, focus (entitlement-gated), pedometer flights-down. The provenance field is what lets detail UI render absent sensors as "not captured on Apple Watch" rather than as failures.
- **Provider code motion = dual target membership, NOT kit absorption.** DispatchKit stays Foundation/SwiftData-only. `LocationProvider`, `WeatherProvider`, `HealthProviders`, `BatteryProvider` move from `App/Sources/Providers/` to a `Shared/Providers/` source group compiled into both `DispatchApp` and `DispatchWatch` via `project.yml` paths (the established `QuickAnswerIntent` dual-membership pattern). Platform conditionals only where the API differs — expected sole case: `BatteryProvider` (`#if os(watchOS)` → WKInterfaceDevice). If a mover needs more than trivial conditionals, it stays app-side and the watch gets a sibling (log the decision). `AudioProvider`, `ConnectionProvider`, `FocusProvider`, `PhotosProvider`, `PedometerProvider`, `VisitObserver`, `WorkoutEndObserver` stay app-only.
- **Sensor toggles are per-device.** `SensorSettings` is UserDefaults-backed and not in the synced store; the watch honors its own instance (default ON), watch settings list toggles for the watch-capable set only. Cross-device toggle sync deferred (schema change out of scope).
- **Watch permissions are the watch's own.** HealthKit + when-in-use location requested contextually at first capture, purpose strings in the watch Info.plist (Apple: frameworks supporting independent watch apps "display their authorization form directly on Apple Watch" — cited in design). Prompt surfaces verified empirically on hardware; ungranted sensors resolve `unavailable` and never block filing.
- **Notifications: scheduling authority stays 100% phone-side.** iOS local notifications forward to the watch automatically per the system's lock-state rules (doc-cited in the design). The watch app schedules NOTHING in v1 — watch-local scheduling would double-prompt (grep-provable: no `UNUserNotificationCenter.add` anywhere in the watch target). Action contract on forwarded notifications (design §v1-scope-5): **Yes/No actions FILE directly on the watch** via the shared filing path — never open-a-screen; **Snooze is a documented no-op on the watch** (dismiss + log; the watch cannot schedule and cannot reach the phone's scheduler — a relay is deferred scope). The action-response routing for forwarded notifications is UNCITED platform behavior: Task 5 carries a mandatory probe (which device's delegate receives `didReceive`; whether action buttons render without watch-side category registration), and the device script verifies on hardware. The design holds under either probe outcome — if responses land phone-side, the phone's existing handler already files and the watch ships tap-to-open (documented downgrade).
- **Quick answer files locally on the watch; the phone reconciles via sync — DEFINITE WORK, not conditional.** Confirmed missing in-tree (2026-07-10): `lastActedAt` is written only by `NotificationScheduler` (action response / marker drain / focus floor), and `RemoteChangeObserver`'s replan callback re-reads the unchanged local value — a synced report does not quiet an in-flight nag chain today. Implement phone-side on remote report arrival, with guards: advance `lastActedAt` from the **report's own timestamp** (never arrival time); **forward-only** (never regress a later value); **ignore historical arrivals** (initial-sync backfill / import / dedupe churn must not act on reports whose timestamps fall outside the live nag chain's window). Kit-tested with fixtures for all three guards.
- **New targets:** `DispatchWatch` (watchOS application, bundle ID `io.robbie.Dispatch.watchkitapp`, `WKApplication`/`WKCompanionAppBundleIdentifier`/`WKRunsIndependentlyOfCompanionApp` Info.plist keys) and `DispatchWatchWidgets` (watchOS WidgetKit extension, `io.robbie.Dispatch.watchkitapp.widgets`). Versions mirror the iOS targets. DispatchKit `Package.swift` gains `.watchOS(.v26)`.
- **Complications:** `accessoryCircular`/`accessoryRectangular`/`accessoryInline`/`accessoryCorner` rendering `WidgetSnapshot` from a read-only shared-store fetch (watch-side mirror of the phone widget architecture). **Watch complications show streak + today count ONLY** — `nextPromptDate` is a planner pass-through fed via the *phone's* app-group defaults and is always nil on the watch (no watch-local scheduling, no defaults sync); layouts must not reserve space for it. Timeline reloads on watch-app save/foreground.
- **v1 watch UI:** quick-answer front and center, then the enabled-question list with minimal per-type inputs (yes/no, choice, number via crown/stepper, text via system dictation/scribble input). No drafts, editing, or visualizations; settings = sync-status line + watch-capable sensor toggles. Reports carry `ReportTrigger` attribution — add an additive `.watch` raw value (v2 export tolerance test, same as `.widget` was).
- **Profiles are the known landmine.** Two new bundle IDs → ASC registration + capability enablement (App Groups, iCloud, **HealthKit, WeatherKit** on the watch app; App Groups only on the widgets) + two new App Store profiles via the ASC API (curl recipe in the session ledger, 2026-07-08; plan 25 documents the pattern) + two new `provisioningProfiles` entries in `scripts/ExportOptionsUpload.plist`. Archive-prove signing BEFORE the commit that depends on it.

## Global Constraints

- Suites green before every commit; scoped commits + push; `git pull --rebase` before starting/pushing. Test-gating absolute: CloudKit/ubiquity/sensors never touched under test args — the watch app takes the same injected-local-store + stubbed-provider path. Do NOT bump the build number (the controller bumps all four targets together at ship time). Additive schema only — this plan's entire schema delta: the two optional provenance fields on `Report` plus the `ReportTrigger.watch` raw value. Accessibility on new watch UI per the plan-17 bar. Every platform-behavior claim in code comments cites a doc URL (four-strikes rule).
