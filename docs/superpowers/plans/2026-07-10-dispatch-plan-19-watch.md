# Dispatch Plan 19: Apple Watch app

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** an independent (companion-style) watchOS app — quick Yes/No answering and minimal report filing from the wrist, complications via WidgetKit accessory widgets, prompt notifications on the watch via automatic forwarding — synced through the existing private CloudKit database.

**Design doc (read first, it carries the citations and the tradeoff record):** `docs/superpowers/specs/2026-07-10-watch-app-design.md`. Tracks GitHub issue #7 — reference it in commits/PRs; do NOT close it from this plan (it covers the whole watch effort).

## Design decisions (decide + log)

- **Independent watchOS app + CloudKit, NOT WatchConnectivity.** The watch is a third device in the already-hardened sync mesh: own local SwiftData store in the watch-side app-group container, `cloudKitDatabase: .private(SyncPolicy.containerIdentifier)`. WatchConnectivity is not used anywhere in v1 (Apple: independent apps "can't rely on the Watch Connectivity framework"; CloudKit is the named sync path — see design doc citations).
- **App Groups are per-device.** The watch app + watch widget extension share the *watch's* `group.io.robbie.Dispatch` container; nothing is shared with the phone through it. No code or comment may imply cross-device container sharing.
- **Notifications: scheduling authority stays 100% phone-side.** iOS local notifications forward to the watch automatically per the system's lock-state rules (doc-cited in the design). The watch app schedules NOTHING in v1 — watch-local scheduling would double-prompt (grep-provable: no `UNUserNotificationCenter.add` anywhere in the watch target). Action contract on forwarded notifications (design §v1-scope-4): **Yes/No actions FILE directly on the watch** via the shared minimal-filing path — never open-a-screen; **Snooze is a documented no-op on the watch** (dismiss + log; the watch cannot schedule and cannot reach the phone's scheduler — a relay is deferred scope). The action-response routing for forwarded notifications is UNCITED platform behavior: Task 4 carries a mandatory probe (which device's delegate receives `didReceive`; whether action buttons render without watch-side category registration), and the device script verifies on hardware. The design holds under either probe outcome — if responses land phone-side, the phone's existing handler already files and the watch ships tap-to-open (documented downgrade).
- **Quick answer files locally on the watch; the phone reconciles via sync — DEFINITE WORK, not conditional.** Confirmed missing in-tree (2026-07-10): `lastActedAt` is written only by `NotificationScheduler` (action response / marker drain / focus floor), and `RemoteChangeObserver`'s replan callback re-reads the unchanged local value — a synced report does not quiet an in-flight nag chain today. Implement phone-side on remote report arrival, with guards: advance `lastActedAt` from the **report's own timestamp** (never arrival time); **forward-only** (never regress a later value); **ignore historical arrivals** (initial-sync backfill / import / dedupe churn must not act on reports whose timestamps fall outside the live nag chain's window). Kit-tested with fixtures for all three guards.
- **New targets:** `DispatchWatch` (watchOS application, bundle ID `io.robbie.Dispatch.watchkitapp`, `WKApplication`/`WKCompanionAppBundleIdentifier`/`WKRunsIndependentlyOfCompanionApp` Info.plist keys) and `DispatchWatchWidgets` (watchOS WidgetKit extension, `io.robbie.Dispatch.watchkitapp.widgets`). Versions mirror the iOS targets. DispatchKit `Package.swift` gains `.watchOS(.v26)`.
- **Complications:** `accessoryCircular`/`accessoryRectangular`/`accessoryInline`/`accessoryCorner` rendering `WidgetSnapshot` from a read-only shared-store fetch (watch-side mirror of the phone widget architecture). **Watch complications show streak + today count ONLY** — `nextPromptDate` is a planner pass-through fed via the *phone's* app-group defaults and is always nil on the watch (no watch-local scheduling, no defaults sync); layouts must not reserve space for it. Timeline reloads on watch-app save/foreground.
- **v1 watch UI:** quick-answer front and center, then the enabled-question list with minimal per-type inputs (yes/no, choice, number via crown/stepper, text via system dictation/scribble input). No sensors, drafts, editing, visualizations, or settings beyond a sync-status line. Reports carry `ReportTrigger` attribution — add an additive `.watch` raw value (v2 export tolerance test, same as `.widget` was).
- **Profiles are the known landmine.** Two new bundle IDs → ASC registration + App Group/iCloud capability enablement + two new App Store profiles via the ASC API (curl recipe in the session ledger, 2026-07-08; plan 25 documents the pattern) + two new `provisioningProfiles` entries in `scripts/ExportOptionsUpload.plist`. Archive-prove signing BEFORE the commit that depends on it.

Verify: build, kit suite, UI suite; manual sim check: file on watch sim → row visible in phone sim app after foreground (shared CloudKit container on sims signed into the same account, or documented as device-script item if sim CloudKit proves flaky — do not fake it). ### Task 4: Complications + notification routing + wrap

- [ ] **Files:** `WatchWidgets/Sources/` (accessory widget: circular/rectangular/inline/corner over `WidgetSnapshot`, read-only store fetch, reload pokes from the watch app on save/foreground), `Watch/Sources/` (UNUserNotificationCenter delegate: `answerYes`/`answerNo` actions FILE via the shared filing path; `snooze` = dismiss + log with a doc-comment explaining why (design §v1-scope-4); plain tap deep-links to the question; NO scheduling calls anywhere in the watch target — grep-provable).

**Contract:** widget timeline renders all four families from a test store, streak/today only (no `nextPromptDate` slot); corner family is watchOS-gated correctly so the iOS widget target is untouched; notification handling's unit-testable part extracted (action identifier → file/no-op/deep-link mapping) and kit-tested. **Mandatory probe (uncited platform behavior, four-strikes rule):** log process + action identifier in both apps' delegates and determine (a) which device's delegate receives `didReceive` for actions on a forwarded notification, (b) whether action buttons render on the watch without watch-side category registration. Record findings as code comments + report; if responses land phone-side, ship tap-to-open on watch and document the downgrade — do not invent a workaround mid-task. Wrap: full suites; device-script list appended to the report (watch↔phone CloudKit sync, forwarding behavior on wrist, forwarded-action routing on hardware, complication refresh, dictation input) — these are sim-unverifiable per the design doc.

Verify: build (warning-free), kit suite, UI suite, archive green. Commit `feat(watch): complications + notification routing` → push. Whole-branch review follows (controller-driven); build bump for all four targets happens at ship time per standing process.
