# Dispatch Plan 31: Calendar-aware prompts (calendar event-end trigger)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** a new prompt-group trigger source (issue #16) — a matching **calendar event ends** → prompt ("How was the meeting?"). Sibling of the shipped visit-arrival trigger (plan 16) and workout-end trigger (plan 12): same `GroupSchedule` event-kind shape, same `gprompt-` identifier family, same planner/budget architecture, with **event-matching rules in the prompt-group editor** (all events / specific calendars / title contains) and EventKit full-access authorization requested lazily from the editor.

**Architecture — one honest divergence from the visit pattern, forced by the platform:** EventKit has **no background wake**. Verified against Apple's docs (2026-07-10): `EKEventStoreChanged` is posted only "while your app is running" ("Updating with notifications", EventKit); nothing in EventKit relaunches a terminated app, and there is no "event ended" callback at all — unlike `startMonitoringVisits()`, whose relaunch delivery is intrinsic. So calendar prompts cannot be posted reactively by an observer at event end. Instead they are **scheduled ahead**: at every replan, a `CalendarEventObserver` fetches matching events across the existing plan windows (today + tomorrow) and the scheduler adds content-addressed `gprompt-<groupID>-<stamp>` requests with `UNCalendarNotificationTrigger`s at each event's END date. That drops calendar groups straight into `replanNow`'s existing machinery — the single removal batch, the focus-filter pre-filter, the `NotificationBudget` allocation, nag chains, and the widget next-prompt readout — which is exactly the planner/budget integration the issue asks for. The observer still exists and still mirrors the visit-arrival lifecycle (launch registration, refresh on group edits and remote-change sync, test-gated, editor-contextual authorization); its "event" is `EKEventStoreChanged` → debounced replan, not the prompt post itself.

**Tech Stack:** EventKit (`EKEventStore`, `requestFullAccessToEvents`, `predicateForEvents(withStart:end:calendars:)`, `EKCalendar`, `.EKEventStoreChanged`), UserNotifications (`UNCalendarNotificationTrigger` — the same trigger type every planned prompt already uses), SwiftData additive optional fields (CloudKit-safe), XcodeGen `INFOPLIST_KEY_*` purpose string.

## Hard constraint for this whole plan

**NO new entitlements** (the plan-16 constraint, still standing: App Store profiles are manually pinned). Verified: on iOS, calendar access is gated by the `NSCalendarsFullAccessUsageDescription` **purpose string** — a plain Info.plist key, not an entitlement (the `com.apple.security.personal-information.calendars` entitlement is macOS-sandbox-only). If anything below turns out to need an entitlement, STOP that item, document, and continue with the rest.

## Design decisions (decide + log)

- **iOS 17+ authorization split (verified against developer.apple.com/documentation/eventkit, "Accessing the event store", 2026-07-10):** reading events requires **full access** — `requestFullAccessToEvents(completion:)`. Write-only access (`requestWriteOnlyAccessToEvents`) cannot read anything: calendar lists return a single virtual calendar and event queries return no results, so `.writeOnly` is *insufficient* for this feature and the hints below must treat it like denied-for-reading. `EKAuthorizationStatus` cases: `.notDetermined / .restricted / .denied / .fullAccess / .writeOnly` (`.authorized` is the deprecated pre-17 case). The purpose string is `NSCalendarsFullAccessUsageDescription`; on iOS 17+ a missing key — or only the legacy `NSCalendarsUsageDescription` — makes the system **auto-deny** the request, so the key lands in `project.yml` in the same task that first calls the request.
- **Authorization is asked lazily and contextually (the plan-16 Always-location precedent), NOT as an onboarding cascade step:** full calendar read is the most sensitive grant in the app and most users will never create a calendar group, so `PermissionCascade` is untouched — no onboarding step, no upgrade top-up. The ONLY ask is from the group editor at the moment the user picks the calendar schedule (the section footnote has just explained why). The request helper follows the cascade/requester house discipline: single-fire continuation through `OneShotResumeGuard` (this codebase shipped a double-resume crash once; EventKit's completion contract is once-only, but the guard costs nothing and the discipline is absolute). Denied / restricted / write-only → the editor and the groups list show an inline "needs calendar access" hint and the group simply doesn't fire — fails quiet, never loud. Doc-required wrinkle: if the `EKEventStore` instance predates the grant, call `reset()` on it after a successful first-time authorization before fetching (per `requestFullAccessToEvents(completion:)`'s reference).
- **Schedule kind `calendarEventEnd` (raw `"calendarEventEnd"`), additive:** `GroupScheduleKind` gains the case; `GroupSchedule` gains `.calendarEventEnd(CalendarEventMatchRule)` — the first event kind with an associated value, because the matching rules are per-group configuration (visit/workout carry none). Unknown-raw forward-compat preserved exactly as today: an older build seeing a synced `"calendarEventEnd"` group resolves it to `.disabled` (never fires, raws preserved on write-back).
- **Match rules — kit value, pure matching:** `CalendarEventMatchRule` = `.allEvents` | `.calendars([String])` (EKCalendar `calendarIdentifier`s) | `.titleContains(String)`. Stored on `PromptGroup` as additive optional fields — `calendarMatchKindRaw: String?` (raws `"allEvents"` / `"calendars"` / `"titleContains"`), `calendarIdentifiersJSON: String?` (JSON `[String]`, the `scheduledTimesJSON` codec pattern), `calendarTitleFilter: String?` — all CloudKit-safe. Leniency: a `calendarEventEnd` group whose match-kind raw is unknown (future rule from a newer build) resolves to `.disabled` — never fires rather than misfires, raws preserved. Degenerate configs fail safe and are prevented at the editor: `.calendars([])` matches nothing; `.titleContains("")` is normalized to `.allEvents` on save (whitespace-trimmed). Title matching is case- and diacritic-insensitive (`range(of:options:[.caseInsensitive, .diacriticInsensitive])`) and lives in a kit pure function so DispatchKit stays EventKit-free and the table is unit-testable.
- **Event semantics (fetch is app-side, filtering is kit-pure):** the observer fetches with `predicateForEvents(withStart:end:calendars:)` per plan window (recurring events expand to occurrences automatically — each occurrence's end is a candidate). The kit's pure `CalendarEventPlanner.fireDates` then: **excludes all-day events** (`isAllDay` — an end-of-day prompt at midnight is noise, and all-day "events" aren't attended things; decided + logged here); keeps events already in progress whose end is still in the future; keeps only ends strictly after `now` and inside `[windowStart, windowEnd)` (the awake window — the asleep replan guard already schedules nothing, matching every other prompt family); sorts ascending and dedupes at minute granularity (the `gprompt` stamp is minute-resolution, so two events ending the same minute produce ONE prompt per group — the content-addressed identifier would collide anyway; the dedupe just makes it deliberate).
- **Budget integration — zero new concepts:** calendar groups enter `replanNow` as a second event-planned family alongside `timerScheduledGroups`: same focus-filter pre-filter (`FocusFilterState.filterPlan` / `allows(groupID:)`), fire counts join the SAME `NotificationBudget.allocate` call as ordinary `groupCounts` entries (groups in sortOrder, allocator clamps in order as designed — a busy calendar day is data-driven load, which is precisely what the allocator exists for), scheduling uses the same `gprompt-<groupID>-<yyyyMMdd-HHmm>` identifiers and `UNCalendarNotificationTrigger` loop, nag chains ride the existing group-stamp path, and the widget next-prompt min() includes them. `GroupPlanner.plan` returns `[]` for `.calendarEventEnd` (its dates come from EventKit, not the distribution machinery) — same as `.workoutEnd`/`.visitArrival`, pinned by test.
- **Scheduler ← observer seam:** `NotificationScheduler` gains a settable `calendarEventSource: (any CalendarEventEndProviding)?` (a one-method `@MainActor` protocol: matched event-end candidates for a rule within a window). `CalendarEventObserver` conforms; DispatchApp wires it after both exist. nil source (tests, or pre-wiring) ⇒ no calendar plans — the scheduler never touches EventKit directly and `replanNow` stays testable exactly as today.
- **Replan freshness:** `.EKEventStoreChanged` (posted only while running — verified, see Architecture) → observer debounces (2s) → replan. Combined with the existing replan triggers (launch, scene-active, group edits, remote-change sync), a schedule stale from *background* calendar edits self-corrects at the next foregrounding. Accepted and documented: worst case a prompt fires for a meeting that was cancelled while Dispatch was suspended — fails quiet, same staleness norm as the focus-filter blob. No BGTaskScheduler (no background modes, no Info.plist task IDs — the plan-16 stance).
- **Trigger attribution:** `ReportTrigger.calendarEventEnd` (additive String raw; older builds fall back to `.manual` via the existing `ReportTrigger(rawValue:) ?? .manual` read — v2 tolerant both ways, same norm as `workoutEnd`/`visitArrival`). userInfo marker `NotificationIdentifiers.calendarEventEndKey` = `"triggeredByCalendarEventEnd"` (`"1"`, mirroring `visitArrivalKey`); the delegate's `didReceive` maps it into the tap-through `SurveyRequest`.
- **No new persistence markers:** schedule-ahead + content-addressed identifiers + the replan's remove-before-add batch already prevent duplicates, so there is no last-handled ledger (unlike the visit/workout observers). `DeleteAllData` needs no new defaults key. The v2 export gains only the three group fields.
- **Watch: out of scope.** Scheduled calendar prompts forward to the watch like every other prompt (they're ordinary `gprompt-` requests); EventKit itself exists on watchOS 2.0+, but nothing watch-side is needed and none is added.
