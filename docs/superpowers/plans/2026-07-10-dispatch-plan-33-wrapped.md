# Dispatch Plan 33: Year in review ŌĆö "Dispatch Wrapped"

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal (issue #18):** a yearly (and monthly) shareable summary ŌĆö "Dispatch Wrapped" ŌĆö composed entirely from stats the kit already knows how to compute: report counts, active days, longest streak, top people/places/tokens, mood trend across the year, notable days, plus the top honest insights. A Wrapped screen reachable from Settings, a shareable image export with the app's theming, and one annual local notification (~Dec 30) that re-engages the user. No server, no comparisons to other users, nothing leaves the device except the image the user explicitly shares.

**Architecture:** a new pure kit aggregation `WrappedStats` in `Sources/DispatchKit/Digest/` (the `DigestStats` precedent: Foundation-only, deterministic, locale-pinned formatting, unit-tested under `swift test`) parameterized over a period (year, or year+month), so monthly comes free from the same compute. App-side: `WrappedView` (the `WeeklyDigestView`/`InsightsView` screen conventions ŌĆö theme background, white-on-theme cards, `.task(id:)` memoized compute), a fixed-size `WrappedShareCard` rendered to a `UIImage` via SwiftUI `ImageRenderer` and handed to `ShareLink`. The annual notification joins `NotificationScheduler.replan()`'s digest block with a new `wrapped-` identifier prefix in `NotificationIdentifiers`, and the tap deep-links through the existing `pendingDigestOpen`-style flag.

**Tech Stack:** Foundation (Calendar math), SwiftUI (`ImageRenderer`, `ShareLink`, `SharePreview`), UserNotifications (`UNCalendarNotificationTrigger` with yearly-repeating month/day components), existing kit types (`Report`, `Question`, `PersonEntity`, `Insight`, `StateOfMindValence`, `ReportStreak`, `PersonResolver`, `Theme`).

## Design decisions (decide + log)

- **Monthly IS in v1 ŌĆö because it is genuinely cheap.** `WrappedStats.compute(reports:questions:people:year:month:calendar:)` takes `month: Int?` ŌĆö nil means the whole calendar year; a value means that single month. The interval filter is the only thing that changes; every downstream aggregation is identical code. The screen exposes it as a segmented "Year / Month" control plus a period picker. Decided over a yearly-only v1 (would save one UI control and zero kit complexity) and over arbitrary date ranges (out of scope, invites filter-subset dishonesty).
- **Period selection is data-driven:** the picker offers only years (and months within the selected year) that contain at least one filed report, newest first, defaulting to the current year. Empty periods can't be selected, so the empty state only appears for a brand-new install (handled with an explainer, the `InsightsView` empty-state convention).
- **WrappedStats composes, never re-invents.** Top people/places/tokens reuse the exact counting conventions of `DigestStats.compute` (tokens vs people by question type, untyped-response catch-all to tokens, places grouped by venue ID else text ŌĆö the ReportsOverview convention), with one deliberate upgrade: people counts resolve through `PersonResolver` + the plan-22 registry (the `InsightsEngine` precedent) so "Angie" and "Angela" merge; `DigestStats` predates the registry and is left untouched. Insights come from `InsightsEngine.compute` over ALL filed reports (never the period subset ŌĆö the engine's logged decision stands; Wrapped displays up to 3 under an "ACROSS ALL YOUR REPORTS" caption so the all-time framing is honest on a period screen).
- **Streaks: `longestStreak(reports:in:calendar:)` is new pure math** (longest run of consecutive report-days that *starts* within the period; a run may finish past the period's end ŌĆö a Dec 20ŌåÆJan 5 streak belongs to the year it started, and is counted at its full length, decided over clipping which would misreport a real streak). `ReportStreak.days` (current streak) is unsuited ŌĆö it anchors at *now*. The new function lives beside `ReportStreak` in `WidgetSnapshot.swift`'s home, `Sources/DispatchKit/Widgets/WidgetSnapshot.swift`? No ŌĆö it is Wrapped-specific aggregation, so it lives as a static on `WrappedStats` (kit-internal helper, tested through `WrappedStats.compute` plus direct unit tests).
- **Mood trend = per-month mean valence buckets:** `monthlyValence: [Double?]` ŌĆö always 12 entries for a year (nil = no answered state-of-mind questions that month), and for a monthly period a single-entry array (the view hides the trend chart when fewer than 2 non-nil buckets exist). Valence mapping via `StateOfMindValence.value` exactly as `DigestStats.meanValence` does. No day-level series in v1 ŌĆö months are the honest grain for a year summary and keep the share card legible.
- **Notable days (deterministic, tie-broken):** `busiestDay: (date: Date, count: Int)?` ŌĆö most filed reports in one local calendar day, earliest date wins ties; `bestMoodDay: (date: Date, valence: Double)?` ŌĆö highest mean valence among days with Ōēź2 answered state-of-mind responses (a single answer is noise, not a "best day"), earliest wins ties. Both nil when no data qualifies.
- **WrappedStats carries presentation-ready primitives, not prose.** Unlike `DigestStats` there is no LLM narrative and no `templateSummary` ŌĆö Wrapped is a visual artifact. Counts, ranked lists (top 5, count-desc then alphabetical ŌĆö the `DigestStats.ranked` ordering), and the two notable days. All number formatting locale-pinned `en_US_POSIX` with explicit grouping (the house convention).
- **Share card is a dedicated fixed-size view, not a snapshot of the screen.** `WrappedShareCard` renders at a fixed 1080├Ś1920 pt 9:16 canvas (Stories/Messages-friendly), themed via `Theme.backgroundHex` ŌåÆ `ThemeColor.color(fromHex:)` with the white-on-theme text treatment the app already uses. It takes `stats` and `theme` as plain values ŌĆö **no `@Environment`/`@Query`** ŌĆö because `ImageRenderer` renders outside the app's environment hierarchy and SwiftData/Observation environment objects would be missing.
- **ImageRenderer usage (verify against current SwiftUI docs at implementation; claims here are plan-level belief):** `ImageRenderer` is `@MainActor`; construct with `ImageRenderer(content: WrappedShareCard(stats:theme:))`, set `renderer.scale` from `@Environment(\.displayScale)` (minimum 2 so exports never come out 1x), read `renderer.uiImage`. The card view must size itself (`.frame(width: 1080, height: 1920)`) ŌĆö `proposedSize` is a fallback, not the primary mechanism. If `uiImage` returns nil (unrenderable content), the share button falls back to disabled with a logged error, never a crash. Verify: exact `scale`/`uiImage`/`proposedSize` API names, and that `ShareLink` accepts an `Image` item with `SharePreview` (iOS 16+ `Transferable` conformance) ŌĆö if sharing a SwiftUI `Image` proves lossy, share the `UIImage` via a `Transferable` wrapper struct instead and record the finding.
- **Share is lazy:** the image renders when the user taps share (or when stats settle), not on every period change. Render is cheap (<50 ms expected) but re-rendering on each segment tap is waste; a `@State` cached image keyed by the period + theme is enough.
- **Notification: `wrapped-annual`, Dec 30 19:00, yearly repeating.** `UNCalendarNotificationTrigger(dateMatching: DateComponents(month: 12, day: 30, hour: 19), repeats: true)` fires once a year (calendar triggers with month+day+hour repeat annually ŌĆö verify with a pending-request check in the sim by inspecting `nextTriggerDate()`). Dec 30 not Dec 31: the user still has a day to act on it inside the year, and it dodges New Year's Eve notification blindness. It schedules in `replan()` directly after the digest block (same "ahead of the asleep guard" placement ŌĆö a year summary is not an awake-window prompt) and joins the standard removal batch via the `wrapped-` prefix.
- **Wrapped notification is ON by default (`wrappedEnabled`, default true) ŌĆö a deliberate divergence from `digestEnabled` (default false), logged here:** the digest fires 52├Ś/year and earns opt-in; Wrapped fires once a year and IS the feature's re-engagement mechanism (issue #18) ŌĆö behind an opt-in it would never fire for the users it exists for. A toggle beside the digest toggle in `NotificationSettingsView` turns it off. `NotificationPrefs` gains the key with a `defaults.object(forKey:) == nil ? true : bool` read (the default-true UserDefaults pattern), and `DeleteAllData`'s preference-key doc list gains `wrappedEnabled`.
- **Notification tap ŌåÆ Wrapped screen** via the `pendingDigestOpen` precedent exactly: `NotificationScheduler.pendingWrappedOpen`, set by the delegate on `wrapped-` prefix match, observed by `ContentView` with the same lock/survey serialization semantics (survey wins, lock wins, Wrapped sheet waits). The sheet hosts `WrappedView` defaulted to the just-ended-or-ending year (Dec 30 tap ŌåÆ that same year).
- **`NotificationIdentifiers.promptSource` needs NO change** ŌĆö unrecognized prefixes already return nil, so `wrapped-annual` can never masquerade as a next prompt. Pinned with a test anyway (the digest identifier has the same pin).
- **Settings placement:** `SettingsView`'s SCHEDULE section gains a `NavigationLink("Wrapped", destination: WrappedView())` directly beneath Insights, identifier `wrapped-link` ŌĆö Weekly Digest / Insights / Wrapped form the natural review-cadence ladder (week / always / year).
- **No widget, no watch, no export-format change in v1.** WrappedStats persists nothing and adds no v2/CSV fields ŌĆö it is a pure read-side aggregation, so there is nothing to round-trip.
